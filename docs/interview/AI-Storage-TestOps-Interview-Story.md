# AI 存储性能 TestOps 面试项目讲解

## 1. 30 秒项目介绍

我基于现有 Spring Boot 测试管理系统，设计并实现了一个 AI 辅助的存储性能 TestOps 平台。它支持自然语言创建测试任务和报告草稿，管理存储样品、测试节点和性能用例，检查节点在线、手机连接和 ADB 状态，模拟分布式节点执行，并将 CDM、AS SSD、FIO 结果标准化入库，最后生成网页预览和 Excel 对比报告。

这个项目的核心不是让 AI 替代测试平台，而是让 AI 作为受控的工作流助手，基于平台已有数据和接口提升测试执行与报告产出效率。

## 2. 2 分钟项目讲解

在真实存储测试工作里，工程师往往需要手动选择节点、确认手机是否连接、执行 ADB 或性能测试脚本、从数据库中捞数据、对比版本差异，并把结果整理到 Excel 报告里。

我把这套流程抽象为三个能力：

- 测试任务模块：负责任务创建、节点检查、用例匹配和执行。
- 测试数据问答模块：负责从标准化测试结果中回答指标排行、版本差异等问题。
- 报告模块：负责从标准化测试结果中组织数据并生成报告。

Agent 负责解析自然语言，并生成测试任务、查询条件或报告草稿。对于测试任务，平台会检查样品信息、节点状态、手机连接、ADB 状态和匹配用例，用户确认后再执行。第一版用 Mock Runner 模拟 CDM、AS SSD、FIO 的测试结果，后续可以替换为真实 ADB 脚本、Jenkins Job 或分布式节点 Agent。

对于报告，系统不要求必须刚跑完一个任务才能生成。它可以基于历史数据、导入数据、上版本数据和竞品数据生成对比报告。报告生成器会计算平均值、差异百分比、warning/fail 阈值，并导出 Excel。

此外，Agent 还可以作为测试数据问答入口。例如用户问“CDM 顺序读最高速率是哪个样品、哪个版本下的”，系统会识别为查询类问题，抽取测试套件、指标和排序目标，通过受控查询工具从标准化结果表中查数据，再返回样品、版本、场景和指标值。

- Pydantic 约束结构化输出

- Mock Runner 保证演示闭环稳定
- Apache POI 生成 Excel 报告
- 测试任务和报告模块解耦
- 报告数据集支持当前版本、基准版本和竞品数据
- 支持基础测试数据问答和指标排行查询
- 节点可用性包含节点状态、手机连接和 ADB 状态
- 预留 RAG + Tool Calling 企业落地架构
- 设计 Eval 集评测意图识别、字段抽取、工具调用、报告计算和总结质量

## 6. 面试可能追问

### 6.1 为什么不让 AI 直接执行 SQL？

因为企业测试数据和执行动作必须可控。AI 可以通过 RAG 理解背景和规则，但实时数据查询、任务创建、报告生成等动作应该通过平台封装好的 API 或工具完成，并受到权限和审计控制。

### 6.2 为什么第一版使用 Mock Runner？

第一版的目标是稳定展示完整平台闭环。Mock Runner 可以模拟分布式节点执行和结果生成。后续只需要替换执行器，就可以接入 ADB 脚本、Jenkins 任务或真实节点 Agent，而不需要推翻任务和报告模型。

### 6.3 为什么要把测试任务和报告模块分开？

真实工作中，报告经常不是跑完一次任务立刻生成，而是需要对比多个版本、多个样品、历史数据和竞品数据。如果报告只能依赖单个任务，就无法覆盖真实企业里的性能对比场景。

### 6.4 LangGraph 放在哪里合适？

LangGraph 更适合后续多步场景，例如缺失字段追问、节点冲突处理、异步等待任务完成、失败重试、报告数据确认和人工审批节点。第一版先用 Pydantic 和规则解析保证稳定，后续再引入 LangGraph 管理复杂 Agent 状态流。

### 6.5 RAG 在这个项目里怎么用？

RAG 适合检索测试规范、数据库说明、用例命名规则、报告模板说明、历史报告结论、常见失败原因和 ADB 故障处理文档。结构化实时数据仍然应该通过平台 API 查询，而不是让 AI 自由拼 SQL。

### 6.6 如果用户不是创建任务或报告，而是问数据怎么办？

Agent 会先做意图识别。如果用户问“CDM 顺序读最高速率是哪个样品、哪个版本下的”，系统会走 `QUERY_RESULT` 或 `ANALYZE_METRIC` 路由，抽取测试套件、指标、排序方式和返回维度，然后调用受控查询工具查询标准化结果表。这样 Agent 不只是固定工作流入口，也能成为测试平台的数据分析助手。

### 6.7 现场 Demo 怎么讲？

我会按三段展示：

- Demo A：自然语言创建测试任务。先调用 `/storage-agent/parse`，再调用 `/storage-tasks/create` 创建任务，最后调用 `/storage-tasks/{id}/run` 触发 Mock Runner。
- Demo B：测试数据问答。调用 `/storage-metrics/query`，回答“CDM 顺序读最高速率是哪个样品、哪个版本下的”。
- Demo C：生成对比报告。调用 `/storage-reports/create` 创建报告数据集，再调用 `/storage-reports/{id}/generate` 生成预览和 Excel，最后通过 `/storage-reports/{id}/download` 下载报告。

这条演示线可以把“自然语言入口、标准化测试结果、数据问答、对比报告和 Excel 导出”完整串起来，既能体现测试平台开发能力，也能体现 AI 应用落地能力。
