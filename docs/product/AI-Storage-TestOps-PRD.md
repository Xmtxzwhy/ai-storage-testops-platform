# AI 存储性能测试管理平台 PRD

## 1. 项目背景

在企业级存储测试工作中，测试工程师通常需要同时操作多个系统和数据源：

- 测试用例库
- 分布式测试节点
- 通过 USB 数据线连接的手机或测试板
- ADB 命令或自动化测试脚本
- 测试结果数据库
- Excel 性能测试报告
- 历史版本报告和竞品对比数据

传统流程中，工程师需要手动选择节点、查找用例、执行脚本、收集数据、对比版本，并把结果复制到固定报告模板中。这类工作重复度高、容易出错，也很难体现现代测试平台的自动化和智能化能力。

本项目计划在现有 Spring Boot 测试管理系统基础上，升级出一套面向存储性能测试场景的 AI TestOps 平台。第一版重点完成任务创建、模拟执行、标准化结果入库和 Excel 报告生成。

## 2. 产品定位

产品名称：

- AI Storage TestOps Platform
- AI 存储性能测试管理平台

产品定位：

面向存储性能测试的 AI 辅助测试管理与报告自动化平台。

系统以自然语言作为入口，但测试任务、执行结果和报告生成仍由平台的结构化数据和受控接口驱动。AI Agent 不直接编造测试数据，而是负责理解用户意图、生成任务或报告草稿、推荐用例和节点、总结结果，并调用平台可信能力完成流程。

## 3. 目标用户

- 测试开发工程师
- 测试平台工程师
- 存储性能测试工程师
- 测试经理
- QA 分析师

## 4. 核心痛点

1. 测试需求需要人工翻译成项目、版本、样品、节点、用例等结构化字段。
2. 节点是否可用不只取决于节点在线，还取决于手机连接和 ADB 状态。
3. 用例选择依赖测试经验，新人很难快速判断该跑哪些测试。
4. 测试执行和报告生成经常不是同一时间发生。
5. 报告可能需要同时对比当前版本、上个版本和竞品数据。
6. Excel 报告填充重复、耗时，且容易出现复制粘贴错误。
7. 企业里通常已经有原有测试平台，AI 应该做提效增强，而不是推翻重做。

## 5. MVP 目标

第一版支持两条相互独立但共享数据的主流程。

### 5.1 测试任务流程

适用于“现在要跑一组测试”的场景。

示例：

> Node-3 上挂了 WM6000 N38B 256G V2.0.4，帮我跑 CDM、AS SSD、FIO 的 clean/dirty 性能测试。

流程：

1. Agent 解析自然语言需求。
2. 平台生成结构化任务草稿。
3. 平台检查样品、节点、手机连接、ADB 状态和用例匹配情况。
4. 用户确认任务草稿。
5. Mock Runner 模拟节点执行。
6. 测试结果写入标准化结果表。

### 5.2 报告生成流程

适用于“已有数据，现在要拉报告”的场景。

示例：

> 帮我生成 WM6000 N38B 256G V2.0.4、V2.0.3 和竞品 2730AB V2.2.5 的性能对比报告。

流程：

1. Agent 识别报告生成意图。
2. 平台生成报告草稿。
3. 用户配置报告数据集。
4. 平台查询标准化测试结果。
5. 平台计算均值、差异百分比、warning/fail 状态和 N/A。
6. 平台生成网页预览数据并导出 Excel。

### 5.3 智能测试数据问答流程

适用于“已有数据，现在想查询或分析”的场景。

示例：

> CDM 顺序读最高速率是哪个样品、哪个版本下的？

流程：

1. Agent 识别用户不是要创建任务或报告，而是在查询测试结果。
2. Agent 抽取查询条件，例如测试套件、指标、排序方式、聚合方式和返回维度。
3. 平台调用受控查询工具，从标准化结果表中查询数据。
4. 系统返回结构化答案，包括样品、版本、场景、指标值、单位和数据来源。
5. 系统可进一步建议生成 Top N 对比表或报告。

## 7. 核心用户场景

### 7.1 创建测试任务

用户输入：

> 项目 WM6000，样品 N38B 256G，版本 V2.0.4，Node-3，跑 CDM、AS SSD、FIO clean/dirty 测试。

系统需要：

- 识别意图：`CREATE_TEST_TASK`
- 提取项目、样品、版本、节点、测试套件和测试场景
- 检查节点和设备状态
- 匹配测试用例
- 创建任务草稿
- 等待用户确认

### 7.2 生成对比报告

用户输入：

> 对比 WM6000 V2.0.4、V2.0.3 和竞品 2730AB，生成 Excel 报告。

系统需要：

- 识别意图：`CREATE_REPORT`
- 创建报告草稿
- 构建当前版本、基准版本、竞品三个数据对象
- 查询结果数据
- 计算差异
- 生成网页预览和 Excel

### 7.3 查询测试指标

用户输入：

> CDM 顺序读最高速率是哪个样品，哪个版本下的？

系统需要：

- 识别意图：`QUERY_RESULT`
- 提取测试套件：CDM
- 提取指标：顺序读
- 提取排序目标：最高速率
- 查询标准化结果表
- 返回最高值对应的样品、版本、场景、测试时间和数据来源

回答示例：

> CDM 顺序读最高的是样品 WM6000-N38B-256G，版本 V2.0.4，在 clean 场景下平均顺序读为 340.46 MB/s，数据来自 2026-07-08 的 CDM 性能测试结果。相比 V2.0.3 的 308.82 MB/s 提升约 10.2%。

### 7.4 分析性能回退

用户询问：

> 为什么 V2.0.4 的 4K 随机写比上个版本低这么多？

后续系统需要：

- 查询当前版本和基准版本指标
- 判断下降发生在 clean、dirty 还是两个场景都有
- 查询相关任务日志
- 查询节点和 ADB 异常事件
- 查询历史报告
- 给出可能原因和建议复测项

### 7.5 批量测试排期

用户输入：

> 明天要测 5 个样品，每个都跑 CDM 和 FIO，优先测 V2.0.4，帮我安排节点。

后续系统需要：

- 查询样品列表
- 查询节点可用性
- 估算测试套件耗时
- 生成排期
- 处理节点冲突和排队

### 7.6 失败任务恢复

用户输入：

> 帮我看一下昨晚失败的任务，能重跑的重跑，不能重跑的列出来。

后续系统需要：

- 分类失败原因
- 自动重试安全失败任务
- 标记 ADB 授权、手机未连接等需要人工处理的问题
- 创建补测任务

## 9. 报告需求

### 9.1 Excel 报告

第一优先级。

第一版模板采用“参考真实报告，但结构简化”的策略。报告包含：

- 基础信息区
- 当前版本数据
- 基准版本或竞品数据
- CDM 结果表
- AS SSD 结果表
- FIO 结果表
- 平均值
- 差异百分比
- warning/fail 高亮
- 人工备注区
- 总结区

### 9.2 网页预览

第二优先级。

网页预览复用报告 JSON 数据，包含：

- 任务或数据集概览
- 指标表格
- 对比表
- warning/fail 异常项
- 导出 Excel 按钮

## 10. 验收标准

MVP 达到以下标准即可验收：

- 用户可以维护样品、节点和测试用例。
- 用户可以输入自然语言并得到结构化任务或报告草稿。
- 系统可以区分创建测试任务和创建报告两种意图。
- 系统可以回答基础测试数据查询问题，例如最高速率、最低速率、Top N、版本差异。
- 系统可以检查节点、手机连接和 ADB 状态。
- 系统可以创建测试任务并调用 Mock Runner。
- Mock Runner 可以写入 CDM、AS SSD、FIO 测试结果。
- 报告生成不依赖刚刚完成的测试任务。
- 报告支持多个数据对象，包括竞品数据。
- 系统可以计算平均值和差异百分比。
- 基准值为 0 或缺失时显示 `N/A`。
- 系统可以导出 Excel 报告。
- 系统可以应用 warning/fail 阈值。
- 项目文档能说明产品范围、架构、数据模型、AI 设计和面试亮点。

## 11. 企业落地扩展方向

在真实企业环境中，通常已经存在测试平台、用例库、节点系统和结果数据库。推荐做法不是推翻原系统，而是在原系统上做 AI 提效增强：

- 将原平台稳定能力封装为 Agent Tools。
- 使用 RAG 检索测试规范、报告模板、数据库说明和故障处理知识。
- 使用 Tool Calling 查询实时数据和执行受控动作。
- 高风险动作必须经过人工确认。

推荐架构：

- RAG 负责知识和上下文。
- Tools 负责查询数据和执行动作。
- 意图路由负责区分任务创建、报告生成、数据查询、异常分析等请求。
- 状态机或 LangGraph 负责多步工作流。
- Eval 集评测意图识别、字段抽取、工具调用、报告计算和总结质量。
